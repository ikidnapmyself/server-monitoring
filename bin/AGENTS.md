# bin — Agent Notes

This file contains **operator-tooling** guidance for working in `bin/`.

## Role in the system

The `bin/` toolchain is the **operator surface**: installer (`install.sh`), interactive CLI (`cli.sh`), update flow (`update.sh`), security posture check (`check_security.sh`), shell aliases. It is invoked from an operator's shell session by a user with login access to the host.

**It is not part of the request-handling pipeline.** It never consumes HTTP, webhook, task-queue, or DB-row input from anywhere other than `manage.py` command output. Any code path here that reads data from the application database for a security-bearing decision should call `manage.py <command>` (which goes through the Django ORM with full validation) rather than `sqlite3 db.sqlite3 "SELECT ..."` or equivalent.

## Key modules

- `bin/install.sh` / `bin/install/` — initial host setup, dependency install, systemd/cron wiring.
- `bin/cli.sh` / `bin/cli/` — interactive operator menus (status, run check, run pipeline, manage notifications, etc.).
- `bin/update.sh` — pulls `origin/main`, regenerates `bin/aliases.sh`, runs migrations.
- `bin/set_production.sh` — production-mode toggle.
- `bin/lib/` — shared shell helpers (`paths.sh`, `update.sh`, `confirm_and_run` patterns).
- `bin/aliases.sh` — generated; do not edit by hand. Regenerated by `update.sh`.

## Security standards (audit-enforced)

Authoritative source: [`docs/plans/2026-05-12-iso-27003-security-audit-notes.md`](../docs/plans/2026-05-12-iso-27003-security-audit-notes.md), `bin/` section.

### Operator-tooling invariants
- **Never consume HTTP / webhook / task-queue input.** If you need data from the running Django app, invoke `manage.py <command>` and parse its output. Do not connect directly to the broker, the result backend, or any inbound HTTP socket.
- **No `sudo`, no setuid, no setcap.** The toolchain runs as the operator's user; it never escalates privilege. If a step genuinely needs root (rare), instruct the operator to re-invoke the script under `sudo` — do not embed `sudo` inside the script.
- **Subprocess spawning is list-form argv.** In shell, prefer `cmd "$arg1" "$arg2"` (quoted, separate words) over `cmd $arg1 $arg2`. Never interpolate user input into a string that is then passed to `eval`, `bash -c`, or `sh -c`.
- **`confirm_and_run` consumes operator stdin only.** Document this assumption at the call site so it stays true. The function reads from the controlling TTY — never from a pipe whose contents could be attacker-controlled.
- **Path-bearing flags are validated by the Django command being invoked.** When a `bin/` script forwards `--path` / `--file` / `--config` to a `manage.py` command, the Django side calls `resolve_safe_path`. Do not pre-resolve the path in shell — pass it through and let the Django side reject it.

### Auto-update trust model
- `bin/lib/update.sh` performs `git fetch origin main` and applies updates from `origin/main`. The operator's chosen `origin` is the trust root.
- Anyone who can push to `origin/main` ships code that the host will execute as the application user.
- Branch protection on `main` (required reviews, status checks) is the actual control — document it in the operator runbook for any deployment that uses auto-update.

### Secrets discipline
- Never `echo` or `printf` API keys, webhook secrets, DB passwords, or `DJANGO_SECRET_KEY` values to stdout/stderr — even during error paths.
- `.env` files generated by `bin/install.sh` have mode `600` and are written via `umask 077` blocks. Preserve that posture in any new generator.
- The interactive CLI surfaces a masked-key display (`{prefix}***`) for `APIKey` — never the full digest, never the raw key (which is unrecoverable after creation anyway).

### Test layout
- Shell tests live under `bin/tests/`. Each new script gets a parallel `bin/tests/test_<script>.sh` exercising at least the happy path and one error path. Use the `test_helper/` framework (excluded from ruff per `pyproject.toml`).

## Audit checks before merging
- [ ] No new code path reads HTTP, webhook, or task-queue input.
- [ ] No new `sudo`, setuid, or setcap invocation.
- [ ] Subprocess spawning is list-form / quoted; no `eval`, no `bash -c "$user_input"`, no `sh -c`.
- [ ] If a generated file holds secrets, it is created under `umask 077` and mode `600`.
- [ ] If the change touches `bin/lib/update.sh` trust assumptions (signed commits, version pinning, etc.), update the "Supply Chain" section of `docs/Security.md` in the same PR.
- [ ] Run `bash bin/tests/run_tests.sh` (or the equivalent harness) and confirm shell tests pass.

## App layout rules

- Shell scripts live at the top of `bin/` or under `bin/cli/`, `bin/install/`, `bin/lib/`.
- Tests live under `bin/tests/` and mirror the script being tested.
- Generated files (`bin/aliases.sh`) are regenerated by `update.sh`; do not edit by hand.
